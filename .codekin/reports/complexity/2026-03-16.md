## Summary

**Overall Complexity Rating: Medium**

The Codekin codebase (~18,000 non-test source lines across ~60 files) is generally well-structured with clear module boundaries in the server layer and dedicated hooks/components on the frontend. The biggest risk concentrations are `SessionManager` (1,710 lines, god-object tendencies) and `App.tsx` (647 lines, orchestrator with 34 React hook calls). The codebase has clearly been refactored incrementally — `SessionManager` already delegates to `ApprovalManager`, `SessionNaming`, `SessionPersistence` — but several files remain oversized. Deepest nesting appears in `ClaudeProcess.handleUserEvent` (4–5 levels) and `ApprovalManager.compactExactCommands` (4 levels). Cyclomatic complexity peaks are in the large switch-dispatchers (`handleMessage`, `handleLine`, `applyMessageMut`) and the multi-strategy approval matchers.

## Coupling & Cohesion Issues

### 1. `SessionManager` — God Object

`server/session-manager.ts` has 1,710 lines and 40+ methods covering: session CRUD, WebSocket join/leave/broadcast, git-diff operations (parseDiff, getDiff, discardChanges), worktree lifecycle, Claude process wiring, approval dispatch, auto-restart scheduling, stall detection, and history buffering. It is imported by `ws-message-handler.ts`, `ws-server.ts`, `session-routes.ts`, `webhook-handler.ts`, `stepflow-handler.ts`, `workflow-routes.ts`, and `commit-event-handler.ts`.

**Suggested fix:** Extract a `DiffManager` class (git ops: `getDiff`, `discardChanges`, `getFileStatuses`) and a `SessionRestartScheduler` (restart/stall logic). Approval dispatch is already delegated to `ApprovalManager` — the thin wrapper methods (`checkAutoApproval`, `derivePattern`, etc.) on `SessionManager` add no value and can be removed so callers reference `ApprovalManager` directly.

---

### 2. `App.tsx` — Prop Drilling Orchestrator

`App.tsx` imports 10 hooks and 12 component files, instantiates 34 React hooks, and passes 20–30 props to `LeftSidebar` and `InputBar`. It acts as a global event bus rather than a compositional root. The three layout branches (workflows, docs-browser, chat) are co-located despite being entirely independent.

**Suggested fix:** Create a `ChatLayout` component that owns `ChatView`, `PromptButtons`, `TentativeBanner`, and `InputBar`. Create a `DocsBrowserLayout` that owns `DocsBrowser` + its `InputBar` variant. Pass the minimum props needed rather than everything from the top-level hook calls.

---

### 3. `InputBar.tsx` — Duplicated Mobile/Desktop JSX

The desktop and mobile rendering paths (`{!isMobile && ...}` / `{isMobile && ...}`) each contain nearly identical permission-mode menu and model-selector dropdown markup (~200 lines combined). Any change to either menu must be made twice.

**Suggested fix:** Extract `<PermissionModeMenu>` and `<ModelMenu>` as standalone components. Pass `orientation` (vertical dropdown vs. compact horizontal) as a prop to control layout differences.

---

### 4. `ws-server.ts` — Mixed Entry Point Concerns

The 536-line entry point mixes: CLI arg parsing, auth token helper functions (`verifyToken`, `verifyTokenOrSessionToken`, `extractToken`), startup health checks, service instantiation, route mounting, WebSocket connection lifecycle, and graceful shutdown. Auth helpers are defined inline rather than reused from `auth-routes.ts`.

**Suggested fix:** Move auth helper functions into `crypto-utils.ts` or a new `server/auth.ts` utility module. Extract startup initialization (workflow engine, commit hooks, version check) into a `startup.ts` function. `ws-server.ts` should only wire together already-constructed services.

---

### 5. `workflow-engine.ts` — Mixed Abstractions

`WorkflowEngine` contains: exported domain types (`WorkflowRun`, `WorkflowStep`, `CronSchedule`), a full cron expression parser, a SQL query builder, and the engine class itself, plus a module-level singleton pattern. These are four separate concerns in one file.

**Suggested fix:** Move types to `workflow-types.ts` (already exists as `stepflow-types.ts` — could extend or co-locate). Move cron parsing to `cron-utils.ts`. Move the query builder inline (it's small) or into a `db-utils.ts`. Keeps `workflow-engine.ts` focused on execution logic only.

---

### 6. `approval-manager.ts` — Static Config as Instance Data

`PATTERNABLE_PREFIXES` and `NEVER_PATTERN_PREFIXES` are large `static readonly Set` constants declared inside the class. They read as config data but are interleaved with logic, making it hard to audit "what is allowed to be patterned."

**Suggested fix:** Extract prefix lists to a `approval-config.ts` constants file. This makes the policy separately auditable and testable without instantiating the class.

## Refactoring Candidates

1. **Extract `DiffManager` from `SessionManager`** (`server/session-manager.ts`, lines ~1375–1590)
   - **Problem:** All git-diff operations (`getDiff`, `discardChanges`, `getFileStatuses`, `execGit`, `execGitChunked`) are embedded in `SessionManager` despite being stateless pure-async operations on a working directory. This makes unit testing `SessionManager` harder and the class harder to navigate.
   - **Approach:** Create `server/diff-manager.ts` as a stateless module (or minimal class) exporting `getDiff(workingDir, scope)`, `discardChanges(workingDir, paths)`. `SessionManager.getDiff` and `discardChanges` become thin delegators.
   - **Effort:** Medium (150+ lines to move, no logic changes needed, update callers)

2. **Break `App.tsx` into layout sub-components** (`src/App.tsx`)
   - **Problem:** 647-line root component mixes orchestration, layout, and event wiring. The three view branches (workflows, docs, chat) each contain non-trivial JSX inline.
   - **Approach:** Extract `<ChatLayout>`, `<DocsBrowserLayout>` components that accept the 5–8 props they actually need, rather than receiving everything from the top level. Extract keyboard shortcuts into `useAppKeyboard()`.
   - **Effort:** Medium (pure restructuring, no logic change)

3. **Deduplicate `InputBar` mobile/desktop menu JSX** (`src/components/InputBar.tsx`)
   - **Problem:** Permission-mode selector and model selector are rendered twice (desktop inline, mobile in a bottom sheet), with nearly identical markup. Any feature change requires dual edits.
   - **Approach:** Create `<PermissionModeMenu onSelect={} currentMode={} compact={} />` and `<ModelMenu onSelect={} currentModel={} compact={} />`. `compact` controls whether items show descriptions (desktop dropdown) or are minimal rows (mobile sheet).
   - **Effort:** Small

4. **Replace `useChatSocket.handleMessage` switch with a handler map** (`src/hooks/useChatSocket.ts`)
   - **Problem:** A 150-line `useCallback` with a 30-case switch containing inline state mutations is hard to test, hard to follow, and grows with every new WS message type.
   - **Approach:** Define a `type MessageHandler = (msg: WsServerMessage, ctx: HandlerContext) => void` and a `const handlers: Partial<Record<WsServerMessage['type'], MessageHandler>>` map. Each case becomes a named function. The `useCallback` becomes a 5-line dispatch trampoline.
   - **Effort:** Medium (logic stays the same, structure changes)

5. **Pre-index cross-repo approval lookups** (`server/approval-manager.ts`, ~line 161)
   - **Problem:** `checkCrossRepoApproval` iterates all repos × all commands/patterns for every approval check. With many repos and frequent tool calls, this is O(R × C) per invocation.
   - **Approach:** Maintain an index `Map<string, Set<string>>` (tool/command → set of repos that approved it), updated in `addRepoApproval` and `removeApproval`. Cross-repo check becomes O(1) for tools and near-O(1) for exact commands.
   - **Effort:** Small–Medium

6. **Extract `SessionRestartScheduler`** (`server/session-manager.ts`, ~lines 778–855)
   - **Problem:** `handleClaudeExit` contains 80 lines of restart-decision logic including API-error detection, worktree cleanup, stall-timer management, and staggered restart delays. This logic is not testable in isolation because it references `SessionManager` state directly.
   - **Approach:** Extract to a `SessionRestartScheduler` class or pure `scheduleRestart(session, exitInfo, callbacks)` function. `SessionManager` wires the callbacks.
   - **Effort:** Medium

7. **Move `ws-server.ts` auth helpers to `server/auth.ts`** (`server/ws-server.ts`, ~lines 72–105)
   - **Problem:** `verifyToken`, `verifyTokenOrSessionToken`, and `extractToken` are defined inline in the entry point, duplicating patterns that also appear in route files. They can't be imported by tests or other modules without importing the full server.
   - **Approach:** Move to `server/auth.ts` and import from there. `createAuthRouter` in `auth-routes.ts` and the WS handler both reference the same source of truth.
   - **Effort:** Small

8. **Split `workflow-engine.ts` types and cron utilities** (`server/workflow-engine.ts`)
   - **Problem:** Domain types, a hand-rolled cron parser, a query builder helper, the engine class, and a singleton export are all in one 669-line file. Adding or auditing any one concern requires scrolling past the others.
   - **Approach:** Move exported types to `server/workflow-types.ts`. Move `parseCronField`, `cronMatchesDate`, `nextCronMatch` to `server/cron-utils.ts`. Engine file becomes ~350 lines and is purely execution logic.
   - **Effort:** Small (mechanical moves)

9. **Extract `approval-config.ts` prefix sets** (`server/approval-manager.ts`)
   - **Problem:** The 50-entry `PATTERNABLE_PREFIXES` and `NEVER_PATTERN_PREFIXES` sets are static config tables embedded inside the class body, making security-policy auditing require reading through class implementation details.
   - **Approach:** Move to `server/approval-config.ts` as exported `const` sets. Import and use from `ApprovalManager`. The policy becomes independently auditable and documentable.
   - **Effort:** Small

10. **Reduce `SessionManager` approval-delegation boilerplate** (`server/session-manager.ts`, ~lines 182–217)
    - **Problem:** `SessionManager` has 7 thin forwarding methods (`checkAutoApproval`, `derivePattern`, `getApprovals`, etc.) that just call the equivalent method on `this.approvalManager`. These methods are only used by `session-routes.ts` and `ws-message-handler.ts`, which already receive a `sessions: SessionManager` parameter.
    - **Approach:** Expose `get approvalManager()` as a readonly accessor on `SessionManager`. Callers access `sessions.approvalManager.checkAutoApproval(...)` directly. Removes 40+ lines of boilerplate forwarding.
    - **Effort:** Small
